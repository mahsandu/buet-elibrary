## Facet: Drupal + Elasticsearch Integration in Academic Libraries

### Key Findings
- The primary integration path for Drupal and Elasticsearch is through the **Search API** module (abstraction layer) paired with the **Elasticsearch Connector** module, which provides the backend adapter. This stack allows Drupal to push content changes to Elasticsearch and query results via Drupal Views or decoupled frontends [^1][^2].
- **Faceted navigation** in Drupal is implemented via the **Facets** module, which works on top of Search API regardless of the backend (Elasticsearch, Solr, or Database). Facet blocks are placed on search pages and filter results dynamically based on indexed fields such as taxonomy terms, dates, or custom metadata [^3][^4].
- The **Elasticsearch Connector** module historically lagged behind Elasticsearch releases; Drupal 8-era tutorials targeted Elasticsearch 5.x, and by 2022 there was active community concern about ES 8.x compatibility with the Drupal module [^5][^6]. Elasticsearch 8.0 introduced breaking REST API changes (removal of types, stricter request parsing, and security auto-configuration) that require connector/library updates [^7].
- **Islandora**—the leading digital repository stack for libraries—uses Drupal as the experience layer but historically relied on **Apache Solr** for discovery. Modern Islandora (8+) keeps Drupal as native CMS with Fedora as optional preservation backend and Solr for search; adapting this to Elasticsearch would require replacing the Solr search module with Search API + Elasticsearch Connector, similar to patterns used by the Tripal Elasticsearch module in biological database projects [^8][^9][^10].
- **Containerized deployment** is well-supported: DDEV, Warden, Platform.sh, and custom Docker Compose stacks all provide recipes for Drupal + Elasticsearch sidecars. For example, Platform.sh uses `composer require drupal/search_api drupal/elasticsearch_connector` and environment-aware `settings.php` to wire clusters automatically [^11][^12][^13].
- For **high-volume library catalogs**, Elasticsearch performance tuning recommendations include: keeping shards between 10–50 GB, using the Bulk API for batch indexing, increasing `refresh_interval` during bulk loads, leveraging filter context for facets, disabling norms on keyword fields, and using Index Lifecycle Management (ILM) for time-based or rolling indices [^14][^15][^16].
- A notable **alternative** to native Search API indexing is the **external-crawler/harvester** pattern, where Drupal remains the frontend/CMS but an external harvester (e.g., DCAT harvester, Logstash, or custom crawler) populates the Elasticsearch index. Norway's data.norge portal used a Drupal 7 frontend with an ELK stack and an open-source DCAT harvester to ingest metadata [^17].
- The **Opensolr** module is a commercial alternative that provides a zero-indexing-load approach for Drupal 10/11: a crawler indexes the site externally, offers vector/AI search, auto-discovered facets, and query elevation without relying on Search API [^18].
- There is **no major library-specific Drupal distribution** that bundles Elasticsearch out-of-the-box; libraries typically build custom stacks using Search API + Elasticsearch Connector + Facets, or adopt Islandora/Solr and swap backends. Cornell's Weill Medicine session notes confirm that Elasticsearch in Drupal lacks a "Solr Search Defaults" equivalent, requiring manual View and facet setup [^19].
- Drupal 11 is technically Drupal 10.3 with deprecated code removed and Symfony 7.1/Twig 3.9 updates. Because Elasticsearch Connector is a contributed module, upgrading to Drupal 11 requires verifying that the module (and its underlying `elasticsearch-php` client) declare compatibility with Drupal 11 and ES 8.x; maintainers often lag behind core releases, so testing and patches are often needed [^20].

### Major Players & Sources
- **Search API module (Drupal.org/project/search_api)**: The canonical abstraction layer for Drupal search backends. Provides indexing, fields, processors, and Views integration. Essential for any Elasticsearch integration via the connector model [^1][^2].
- **Elasticsearch Connector (Drupal.org/project/elasticsearch_connector)**: The main backend module linking Search API to Elasticsearch. Uses the official Elasticsearch PHP client. Supports cluster management, index mapping, and fuzziness settings. Maintained by Nikolay Ignatov and others [^1][^6][^11].
- **Facets module (Drupal.org/project/facets)**: Provides the UI and logic for faceted navigation on Search API indexes. Works with any backend and is the standard for Drupal discovery layers [^3][^4].
- **Islandora (islandora.ca / GitHub)**: The dominant open-source digital repository framework for academic libraries. Combines Drupal, Fedora, and Solr. Offers microservices (Crayfish), batch ingestion (Workbench), and linked-data export (JSON-LD, OAI-PMH). It is the primary reference architecture for library discovery on Drupal, though Solr-centric [^8][^9][^10].
- **OpenSense Labs / Red Crackle**: Drupal agencies that published canonical integration tutorials and case studies (e.g., Produce Market Guide faceted search with React + Elasticsearch + Search API). Good reference for architecture patterns and decoupled search UIs [^2][^4].
- **Tripal Elasticsearch**: A biological-database community extension that demonstrates Drupal + Elasticsearch for large scientific datasets, including customized advanced searches and indexing. Relevant because it shows how a domain-specific Drupal community can replace Solr with Elasticsearch [^10].
- **Elasticsearch (Elastic.co)**: Search engine vendor. Elasticsearch 8.x introduced REST API compatibility headers, security auto-configuration, and removal of mapping types. These affect client libraries and therefore Drupal connectors [^7][^21].
- **DDEV / Platform.sh / Warden**: Tooling and PaaS vendors that provide containerized Drupal environments with Elasticsearch add-ons, simplifying local development and production deployment [^11][^12][^13].

### Trends & Signals
- **Shift from Solr to Elasticsearch in non-library sectors**: Drupal commerce and media sites increasingly adopt Elasticsearch for real-time analytics, log aggregation, and vector search. This pressure is slowly pushing contributed modules to support newer ES versions [^2][^14].
- **Container-first deployment**: Modern Drupal hosting (DDEV, Platform.sh, Kubernetes Helm charts) treats Elasticsearch as a standard sidecar service, making the stack more accessible for library infrastructure teams [^11][^12][^13].
- **Decoupled / progressively decoupled discovery layers**: Case studies show React or Vue frontends querying Elasticsearch directly via Searchkit or custom APIs, while Drupal handles content management. This pattern is attractive for library discovery interfaces that need high interactivity [^2].
- **Vector/semantic search emerging**: Opensolr and Elasticsearch 8.x native vector/kNN features are creating demand for AI-assisted search in Drupal. Academic libraries may follow this trend for semantic article/citation discovery [^18][^21].
- **External harvesting over native indexing**: For large metadata catalogs, external harvesters (OAI-PMH, DCAT, RDF) that write directly to Elasticsearch are gaining traction because they remove indexing load from Drupal and allow pre-processing (e.g., Norway's data.norge portal) [^17].
- **Islandora staying Solr-first but Fedora-optional**: Discovery Garden and the Islandora community have made Fedora optional and are investing in Solr-backed advanced search and IIIF. There is no official Elasticsearch migration, but the modular architecture (Search API + custom modules) makes it theoretically possible [^8][^9].

### Controversies & Conflicting Claims
- **Elasticsearch Connector vs. Elastic Search module**: Two competing approaches exist. Search API + Elasticsearch Connector is the "Drupal-native" way but suffers terminology mismatches (Search API "index" vs. Elasticsearch "type"). The standalone Elastic Search module uses the official ES DSL and conventions but is less integrated with Drupal Views and Search API. Most agencies recommend the Search API route for standard CMS use, while the Elastic Search module is better for decoupled/custom applications [^1][^2].
- **ES 8.x compatibility uncertainty**: An Elastic.co forum post from 2022 asked whether the Drupal Elasticsearch Connector was compatible with ES 8.x; the response only warned that ES 7.10 was EOL and did not answer the module question, indicating a documentation/support gap. As of recent Drupal 10/11 guides, authors still recommend verifying connector versions carefully because ES 8.x removed types and changed security defaults [^6][^7][^20].
- **Drupal core search vs. external engine**: Some library case studies (e.g., Colgate University) emphasize that Drupal's built-in database search is sufficient for small sites, but the literature consistently warns that for large catalogs, database search is "not very efficient" and dedicated engines (Solr/Elasticsearch) are necessary [^19][^22].
- **Performance claims vs. reality**: Elasticsearch vendors and Drupal agencies often claim "near real-time" search and massive scalability, but practitioners note that indexing speed degrades as indices grow beyond 50–100 GB, and that proper shard sizing, refresh intervals, and heap tuning are mandatory. Unoptimized Drupal + ES stacks can still perform poorly [^14][^15][^16].
- **Islandora's Solr lock-in**: Islandora documentation and community resources focus almost exclusively on Solr. While the Islandora 8 architecture is "native Drupal" and could theoretically use any Search API backend, the lack of an official Elasticsearch module means libraries wanting Islandora features with ES must custom-develop or forego Islandora-specific search derivatives [^8][^9].

### Recommended Deep-Dive Areas
- **Elasticsearch 8.x Connector readiness for Drupal 11**: Because BUET is targeting Drupal 11 and ES 8.x, the exact version matrix of `drupal/elasticsearch_connector`, `elasticsearch/elasticsearch-php`, and Drupal 11 core needs to be tested. If the contributed module is not yet certified, patching or using a custom client wrapper may be required. This warrants immediate prototyping.
- **External harvester vs. native indexing architecture**: For a library catalog with hundreds of thousands of MARC/DC records, pushing every field through Drupal's Search API into ES may be slower and more fragile than having an external Python/Go harvester populate ES and letting Drupal query it via a lightweight custom module or JSON:API. This is a strategic architecture decision.
- **Faceted search performance at scale**: Vinted's engineering case study showed that multi-select facets require separate ES aggregation requests per facet and caching to maintain low latency (~120ms P99). A library catalog with many facet categories (author, subject, date, format, language) will need similar optimization [^23].
- **Islandora adaptation feasibility**: If BUET wants Islandora-like features (digital preservation, IIIF, linked data), a deep-dive should assess whether Islandora's advanced_search module and Crayfish microservices can be decoupled from Solr, or whether BUET should build a custom digital repository module on top of Drupal 11 + Elasticsearch instead.
- **Containerized deployment patterns**: DDEV and Docker Compose recipes for Drupal 11 + ES 8.x need to be collected and tested, especially with custom ES plugins (e.g., ICU analysis for Bengali text) and Kibana for monitoring. A Helm chart for Kubernetes would be a long-term deliverable.
- **MARC/Dublin Core field mapping**: How library metadata fields map to Elasticsearch analyzers (keyword vs. text, date formats, multi-language) is critical. The mapping design directly affects faceted navigation relevance and sorting. This requires collaboration with cataloging experts.
